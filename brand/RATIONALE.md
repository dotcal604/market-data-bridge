# Brand Rationale — market-data-bridge

## Why This Mark

The Bridge Grid was chosen through a 9-phase process: discovery, strategy, divergent ideation (5 concepts, 15+ variants), first-round assets, internal critique (5 perspectives), simulated UAT (3 cohorts), two rounds of iteration, and final decision.

### What it represents

The mark is a 3x3 grid read left-to-right:

1. **Left column (filled gray)**: Multiple data sources with varying characteristics — IBKR real-time feeds, Yahoo Finance delayed data, CSV/XLSX broker exports. Different gray tones represent different source types.

2. **Center column (outlined)**: The bridge layer — normalization, schema alignment, routing. Outlined because it's infrastructure: present but transparent. Data passes through; the bridge doesn't add its own opinion.

3. **Right column (emerald with dots)**: Enriched output — 14 computed features, ensemble scores, normalized schemas, tool-ready data. Emerald because it's the codebase's own color for "positive signal." Dots because the output contains data points, not just forwarded bytes.

### What was considered and rejected

- **"The Join" (convergence lines)**: Architecturally honest but too many visual elements. Collapsed at favicon sizes. More illustration than mark.
- **"The Ensemble Grid" (3-model scoring)**: Distinctive but branded a subsystem (eval engine) instead of the whole product.
- **"The Bridge Span" (structural beam)**: Too literal — could be any integration platform.
- **"The Protocol Mark" (interlocking shapes)**: Too abstract to decode without context.
- **"The Enrichment Pipeline" (thin→thick)**: Weak at small sizes, generic "transformation" metaphor.

### Why it works

1. **Specificity**: The three-column source→bridge→output progression maps exactly to the product architecture. This is not a metaphor — it's a diagram.
2. **Scalability**: A 3x3 grid is inherently pixel-aligned. It works at 16x16 (just 9 tiny squares) and 512x512 (full detail with dots and opacity variations).
3. **Monochrome safety**: Filled → outlined → filled-with-dots reads in grayscale without any color.
4. **Codebase-native**: The emerald accent (#10b981) is already the product's "positive signal" color. The slate palette matches the dashboard's dark theme. Nothing was imposed from outside.
5. **Audience-appropriate**: Engineers who live in terminals and grids see structured data, not decoration. The monospace wordmark reinforces this.

### Known tradeoffs

- The mark doesn't scream "bridge" without the name — it reads more as "data pipeline" or "schema transformation." This is acceptable because the name provides the framing, and the mark provides the structure.
- The three-color source tones (varied grays) are subtle. In some contexts, they may read as identical. This is by design — the sources are different but equivalent in importance.
- MDB as shorthand is deliberately suppressed in the primary identity. It's available for UI badges and CLI output but never replaces the full name.
