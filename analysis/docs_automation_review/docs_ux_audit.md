# Docs UX / Information Architecture Audit

## Branch A: `automate-docs-generation` (Docusaurus)

### Site Structure
```
/ (landing page — generic project intro)
├── /docs/api/          ← TypeDoc-generated TypeScript module docs
├── /docs/architecture/ ← Hand-written Mermaid architecture page
└── /docs/previews/     ← Static table of frontend page names (no screenshots)
```

### Assessment

**Does the top-level structure fit this repo?**
No. The three nav items are "API Reference" (TypeScript internals), "Architecture" (Mermaid diagrams), and "UI Previews" (empty table). None of these serve the primary users of this repo:
- MCP tool users (Claude Code/Desktop) → no MCP reference
- REST API integrators (ChatGPT Actions) → no OpenAPI/REST docs
- Operators/deployers → no deployment or runbook content

**Is the IBKR + MCP + REST identity clear?**
No. The landing page says "Auto-generated API docs, architecture diagrams, and UI previews" — this could describe any project. The repo's identity as an IBKR trading bridge with MCP + REST interfaces is buried.

**Are the highest-value docs obvious?**
No. The highest-value docs for this repo are:
1. MCP Tool Reference (152 tools) — **missing**
2. REST API Reference (135 actions) — **missing**
3. Deployment/Operations Guide — **missing**
The docs that ARE present (TypeDoc class docs, Mermaid diagrams) are low-value for the target audience.

**Would a new user find the right thing quickly?**
No. A user arriving at the docs site would see TypeScript module documentation and think this is a library, not an MCP server.

**Strongest parts:** Architecture Mermaid diagrams are well-written and informative.
**Weakest parts:** Everything else — wrong docs for the audience.

## Branch B: `docs-automation-mvp` (Zero-dep generator)

### Site Structure
```
/ (landing page with 4 nav cards)
├── /guide/
│   ├── overview.html     ← docs/01-PROJECT-OVERVIEW.md
│   ├── architecture.html ← docs/02-ARCHITECTURE.md
│   ├── api-reference.html← docs/03-API-REFERENCE.md
│   ├── deployment.html   ← docs/04-DEPLOYMENT-GUIDE.md
│   ├── runbook.html      ← docs/05-RUNBOOK.md
│   └── user-guide.html   ← docs/06-USER-GUIDE.md
├── /api/                 ← Redoc-rendered OpenAPI spec (135 actions)
├── /mcp/                 ← Auto-extracted MCP tool reference (152 tools)
└── /architecture/        ← Embedded architecture-diagram.html
```

### Assessment

**Does the top-level structure fit this repo?**
Yes. The four sections (Guide, REST API, MCP Tools, Architecture) directly serve the repo's primary users. MCP users find tool docs, REST integrators find API docs, operators find deployment guides, and everyone can see the architecture.

**Is the IBKR + MCP + REST identity clear?**
Yes. The landing page says "IBKR trading bridge with MCP + REST interfaces, 3-model AI ensemble, and real-time dashboard." This accurately describes the project. The nav prominently features both REST API and MCP Tools.

**Is Divoom properly scoped?**
Yes. Divoom tools appear in the MCP reference under their own "Divoom" category (auto-categorized by prefix). They are not overemphasized — just listed alongside other tool categories. No dedicated Divoom section exists, which is appropriate.

**Are the highest-value docs obvious?**
Mostly. REST API (Redoc) and MCP Tools (extracted reference) are both in the top-level nav. The guide section provides depth. Architecture is accessible but not intrusive.

**Would a new user find the right thing quickly?**
Yes. Landing page cards point directly to the 4 sections. A Claude Code user clicks "MCP Tools" and sees 152 tools organized by category. A ChatGPT integrator clicks "REST API" and sees the full OpenAPI spec rendered by Redoc.

**Would a contributor know where to update docs?**
Mostly. The `source-info` section on the landing page explains: "MCP tool documentation is extracted directly from `src/mcp/server.ts` tool registrations. REST API docs are rendered from the OpenAPI spec generated by `src/rest/openapi-gen.ts`." A contributor adding a new MCP tool just adds it to `server.ts` and the docs auto-update.

Guide content requires updating the `docs/01-06*.md` files, which is the existing workflow.

**Is the architecture section useful?**
Yes. It embeds the existing `architecture-diagram.html` which is an interactive visualization. No new architecture content was created — it reuses what exists.

**Are frontend previews included?**
No, and this is correct. Branch B does not include frontend previews. The frontend is a Next.js dashboard that changes frequently. Static screenshots would go stale immediately.

### Strongest Parts
1. **MCP Tool Reference**: 152 tools, organized into 20+ categories, with parameter tables. Auto-generated from source — this is the single highest-value section.
2. **REST API via Redoc**: Professional interactive API explorer. Zero maintenance — just copy the OpenAPI spec.
3. **Clean navigation**: 4 sections, clear purpose, no clutter.
4. **Source attribution**: Every page says where its data comes from.

### Weakest Parts
1. **Missing Mintlify MDX content**: 28 rich documentation files in `docs/mcp/*.mdx`, `docs/workflows/*.mdx`, and `docs/frontend/*.mdx` are ignored. These contain hand-written workflow guides, trading examples, and page-by-page frontend docs. Some of this content (especially `docs/workflows/`) is high-value.
2. **Guide page rendering quality**: The custom markdown parser will produce imperfect HTML for complex markdown features (nested lists, blockquotes). The 60K-line guides likely exercise edge cases.
3. **Dark-theme-only**: The generated site has a dark theme with no light mode toggle. This is fine for a trading tool audience but may reduce accessibility.
4. **No search**: No search functionality. For 152 MCP tools + 135 REST actions + 6 guide pages, search would be valuable. (Reasonable to defer for MVP.)
5. **Hardcoded "135 agent actions"**: Landing page card uses a static count instead of deriving it from the OpenAPI spec.

### Missing Sections (Potential Future)
- Workflow guides from `docs/workflows/*.mdx`
- Frontend page documentation from `docs/frontend/*.mdx`
- Changelog / release notes
- Search
- Light/dark toggle

### Unnecessary Sections
None. All 4 sections serve real users.
