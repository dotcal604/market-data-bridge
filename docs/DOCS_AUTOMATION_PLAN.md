# Docs Automation Plan — Market Data Bridge

## Repo Audit Summary

### What exists today

| Asset | Location | Status |
|-------|----------|--------|
| Mintlify docs structure | `docs/`, `docs.json` | MDX files structured for Mintlify (paid hosted service), never deployed |
| OpenAPI spec generator | `src/rest/openapi-gen.ts` | Generates OpenAPI 3.0 JSON from `actionsMeta` in `src/rest/agent.ts` (135 actions) |
| Static OpenAPI JSON | `openapi-chatgpt.json` | Generated artifact, stale-risk (not regenerated in CI) |
| MCP tool definitions | `src/mcp/server.ts` | ~150 tools with names, descriptions, Zod schemas — all inline, no extraction |
| Architecture diagram | `architecture-diagram.html` | Rich interactive HTML, standalone |
| Numbered docs series | `docs/01-PROJECT-OVERVIEW.md` through `docs/06-USER-GUIDE.md` | Comprehensive, manually maintained |
| Root-level docs | `README.md`, `SYSTEM_CARD.md`, `ROADMAP.md`, `CONTRIBUTING.md`, etc. | Good content, not connected to any docs site |
| Existing CI workflows | `.github/workflows/` | Build, test, Docker — no docs workflow |

### Key findings

1. **No docs deployment**: The Mintlify structure exists but was never deployed (requires paid subscription)
2. **MCP docs are manually maintained**: `docs/mcp/*.mdx` files are hand-written, not generated from the actual tool registrations in `server.ts`. High staleness risk with 150+ tools.
3. **OpenAPI JSON is stale-risk**: `openapi-chatgpt.json` is checked in but not regenerated in CI
4. **Duplicate docs content**: Root-level markdown files overlap with `docs/` numbered series
5. **Architecture diagram is standalone**: The HTML diagram is useful but not integrated into any docs site

## Chosen Architecture

**Zero-dependency static site generated by a single Node.js script.**

### Why not Docusaurus / Mintlify / other framework?

- **Docusaurus**: Adds ~200MB of dependencies, React build pipeline, MDX compilation. Overkill for a project that needs to show API docs, MCP reference, and a few guide pages.
- **Mintlify**: Requires paid subscription and external hosting. Can't deploy to GitHub Pages.
- **Jekyll**: Would work with GitHub Pages native support, but MDX files need conversion, and we'd lose the ability to auto-generate MCP docs from source.
- **Custom Node.js script**: Zero new dependencies. Uses markdown-to-HTML conversion with a simple template. Can extract MCP tools from source. Can embed Redoc for OpenAPI. Easy to understand, maintain, and extend.

### Why this is the right MVP

- **Smallest worthwhile scope**: Generates a real, deployable docs site with source-of-truth-aware content
- **No new dependencies**: Uses only Node.js built-ins (fs, path, child_process)
- **Deterministic**: Same source → same output, every time
- **Maintainable**: One script file, one workflow, one output directory
- **Source-of-truth-aware**: MCP docs come from actual tool registrations, not manual copies

## Source Inputs

| Source | What it provides | Generation method |
|--------|-----------------|-------------------|
| `src/mcp/server.ts` | MCP tool reference (names, descriptions, param schemas) | Regex extraction from `server.tool()` calls |
| `src/rest/openapi-gen.ts` | OpenAPI 3.0 spec | Imported and invoked at build time (or use checked-in JSON) |
| `openapi-chatgpt.json` | REST API reference | Rendered via Redoc standalone HTML |
| `docs/01-*.md` through `docs/06-*.md` | Guide content | Converted to HTML pages |
| `architecture-diagram.html` | Architecture visualization | Embedded as iframe or linked |
| `README.md` | Project overview | Converted to landing page |

## Generation Flow

```
npm run docs:generate
  └─ node scripts/generate-docs.mjs
       ├─ 1. Parse src/mcp/server.ts → extract tool registrations
       ├─ 2. Copy openapi-chatgpt.json → docs-site/openapi.json
       ├─ 3. Generate Redoc HTML page → docs-site/api/index.html
       ├─ 4. Generate MCP reference HTML → docs-site/mcp/index.html
       ├─ 5. Convert docs/*.md → docs-site/guide/*.html
       ├─ 6. Embed architecture diagram → docs-site/architecture/index.html
       └─ 7. Generate index.html landing page
```

## CI/CD Flow

```yaml
# .github/workflows/docs.yml
on:
  push:
    branches: [main]
  workflow_dispatch:

jobs:
  build-docs:
    runs-on: ubuntu-latest
    steps:
      - checkout
      - setup node
      - npm ci
      - npm run docs:generate
      - upload pages artifact (docs-site/)
  deploy:
    needs: build-docs
    uses: actions/deploy-pages
```

## First-Time Setup vs Steady-State

### First-time setup
1. Enable GitHub Pages in repo settings (Source: GitHub Actions)
2. Push to main — workflow runs automatically
3. Site appears at `https://<owner>.github.io/market-data-bridge/`

### Steady-state
- Any push to `main` regenerates and deploys docs
- MCP tool changes in `server.ts` are automatically reflected
- OpenAPI spec is copied from checked-in JSON (regenerated separately via existing tooling)
- Guide content updated by editing `docs/0X-*.md` files

## MVP vs Deferred Scope

### MVP (this implementation)
- [x] MCP tool reference auto-generated from source
- [x] OpenAPI/REST docs via Redoc
- [x] Guide pages from existing markdown
- [x] Architecture diagram embedded
- [x] GitHub Actions workflow for generation + deployment
- [x] Simple landing page with navigation

### Deferred
- [ ] TypeDoc for internal API (low value vs MCP/REST docs)
- [ ] Frontend page previews (Storybook or screenshots — high complexity, low urgency)
- [ ] Search functionality (could add Pagefind later)
- [ ] OpenAPI regeneration in CI (requires build step; current flow of checking in the JSON works fine)
- [ ] Versioned docs (no versioning need yet)
- [ ] Custom domain setup
- [ ] Dark/light theme toggle (ships dark-only for now)

## Key Tradeoffs

| Decision | Chosen | Alternative | Rationale |
|----------|--------|-------------|-----------|
| Site framework | Custom script | Docusaurus | Zero dependencies, full control, matches repo complexity |
| Markdown rendering | Simple regex-based | markdown-it/marked | No npm dependency needed for basic content |
| MCP docs source | Regex parse server.ts | Runtime extraction | No build/runtime dependency, works in CI without TWS |
| OpenAPI rendering | Redoc CDN | Swagger UI | Cleaner output, single HTML file, better for reference docs |
| Architecture diagram | Embed existing HTML | Mermaid conversion | The HTML diagram is already excellent; don't rebuild it |
| Frontend previews | Deferred | Storybook/screenshots | High setup cost, low docs value in v1 |
